Open Engine — Architecturev2

SECTION 0

TL;DR

Linear is the shared task queue. Every task is a Linear issue whose title carries brackets ([agent instructions][<agent-code>][task] …) and whose description is a 7-part task record. Issues carry the agent-instructions label. Each runtime executes an 11-step queue runner that checks the ledger, does standing preflight, resumes blocked/held issues, follows up on delegations, and otherwise claims the oldest eligible Agent Todo issue — then stops after exactly one task. Every state change is recorded with a receipt (one of 15 canonical tokens) as a comment + label + status patch. Each agent owns one AGENT STATUS ledger comment it updates in place every run.

We start with one task, one agent, one blocker rule, one receipt and run all 4 canonical smoke tests before scaling.

SECTION 2

The 7-Part Task Record

  1. Requester — who asked (human or agent)
  2. Desired outcome — what "done" looks like, as an outcome
  3. Sources — refs, docs, links, prior issue IDs (state that travels)
  4. Acceptance criteria — testable definition of done (checkboxes)
  5. Boundaries — what is NOT in scope
  6. Blocker rule — what to do when stuck (on-Linear → BLOCKED; off-Linear → HOLD)
  7. Receipts — filled by agents as work progresses

Title format (exact)

[agent instructions][<agent-code>][task] <short outcome>

The second bracket is the runtime that may claim it (enforced by runner step 7). [all agents] is used only for standing/ledger/directory issues, never for tasks.

SECTION 4

The Queue Runner — 11 Steps

The exact prompt each runtime executes every run. One run handles exactly one task issue.

  1. Open ledger, mark checking. Find your AGENT STATUS comment; update in place — Last queue result = checking, Last heartbeat = now.
  2. Mandatory standing preflight. Compare local context versions vs standing issues. Install/adapt locally first, then leave AGENT APPLIED only after actually installing locally.
  3. Optional standing skill preflight. Check only skills already installed/subscribed. Apply same-scope updates automatically (AGENT SKILL UPDATED). Do not browse/install unapproved skills during routine runs.
  4. Check AGENT HUMAN HOLD issues. If one now shows AGENT HUMAN ANSWERED → move to Agent Working, leave AGENT RESUMED, finish it, stop.
  5. Check AGENT BLOCKED issues. If one now has its answer on the same issue → move to Agent Working, leave AGENT UNBLOCKED then AGENT RESUMED, finish it, stop.
  6. Check delegated issues. Leave AGENT FOLLOW-UP on your issue if any delegated issue's state changed.
  7. Claim the oldest eligible Agent Todo issue. Eligible = agent-instructions label + [agent instructions] in title + your agent code as the second bracket. Move to Agent Working, leave AGENT CLAIMED, re-read the issue.
  8. Do only scoped work. Done, no human judgment → AGENT DONE + Agent Done. Done, needs review → AGENT DONE + Agent Review.
  9. If blocked: answer on Linear → AGENT BLOCKED (one question) + Agent Needs Input + ledger "blocked ID", stop. Answer in human's thread → AGENT HUMAN HOLD + Agent Needs Input + ledger "holding ID", stop.
  10. If execution fails unexpectedly → AGENT FAILED with last safe step + retry count.
  11. Update ledger and stop after exactly one task issue.

Eligibility (step 7), formalized

An issue is eligible iff all hold: label set includes agent-instructions; title contains [agent instructions]; the title's second bracket equals the runtime's agent code; status is Agent Todo. Encoded in is_eligible_for_claim() (self-tested).

SECTION 6

Standing Skills Architecture

Optional standing skills are discoverable from the directory but NOT auto-installed.

  • First install requires human approval in that runtime's agent thread.
  • Approval subscribes the runtime to future same-scope updates (auto-applied at step 3).
  • Scope expansion needs fresh approval.

SECTION 7

Private Context (per-runtime SKILL.md)

Each runtime keeps a local private context file (not in the repo, not in Linear): e.g. ~/.hermes/skills/open-engine/SKILL.md for Eos. It holds: engine version, agent code, Linear team/project, label, allowed sources, status ledger issue ID, optional skill directory ID, the runtime's own AGENT STATUS comment ID, installed/subscribed skills, and safety boundaries. Plus a private Standing setup issue listing what to install.

Hyperagent mapping

For Hyperagent agents (Bolt), the equivalent is a pinned memory + this repo's docs. The SKILL.md format is the canonical structure; map it onto whatever private-context mechanism the runtime provides.

Hard rules (in each runtime's private context)

  • One task issue per run.
  • Never post a fresh AGENT STATUS — edit in place.
  • Never claim an issue whose second bracket isn't your code.
  • Patch status on every receipt that implies a status change (Paperclip #1).
  • Do not browse/install unapproved skills on routine runs.
  • Do not cross the task record's §5 Boundaries.
  • Auth by creator, not assignee (Paperclip #3).

Handoffs & parent/child (Paperclip #4)

A handoff = one agent's AGENT DONE becomes another's Agent Todo. The reviewer (Eos) creates the follow-on, linking the completed one in §3 Sources. Parent/child use Linear parentId + blocks/blockedBy explicitly — parents close explicitly at review.

SECTION 12

Deployment & Operations

Not a deployed service

Open Engine is: (1) a Linear configuration (project + 6 statuses + agent-instructions label + 14 receipt labels + 2 standing issues), (2) this GitHub repo, (3) a private SKILL.md per runtime + the queue-runner prompt. No server, no cron, no database for the engine itself. Linear is the database. The runtimes are the executor.

Create project + labels + ledger + directorypython3 scripts/linear/setup.py
Create the 6 statusesLinear UI (Workflow → Statuses) — MCP cannot
Create a taskEos fills templates/task-record.md; creates via linear__save_issue
Review blocked/heldFilter: labels AGENT BLOCKED / AGENT HUMAN HOLD
Review completedFilter: status Agent Review
Audit a tasklinear__list_comments — the thread is the audit log
Add a runtimeCreate its SKILL.md; embed queue-runner prompt; ensure tools; it creates its AGENT STATUS on first run

Maintenance

Quarterly: review last 10 tasks (7 parts right? receipts left?); confirm 14 labels exist & used; confirm each runtime's SKILL.md + runner prompt current; confirm each runtime has exactly one AGENT STATUS comment; retire/fix drifted runtimes.
